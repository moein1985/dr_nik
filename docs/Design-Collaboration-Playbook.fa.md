# دستورالعمل همکاری طراحی لندینگ

## هدف این سند
این سند برای همکاری بین تیم توسعه و طراح UI تهیه شده تا:
- خروجی طراحی دقیق، قابل اجرا و سریع تحویل شود.
- دوباره کاری کم شود.
- نیازهای فنی از ابتدا شفاف باشد.
- کیفیت نهایی لندینگ با نیاز کارفرما هم راستا بماند.

---

## نقش ها و مسئولیت ها

### طراح (متخصص Photoshop)
- تبدیل نیازهای کارفرما به طراحی UI حرفه ای و قابل پیاده سازی.
- ارائه فایل های منظم، لایه بندی شده و خروجی assets.
- رعایت محدودیت های فنی اعلام شده توسط توسعه.
- ارائه نسخه های بازنگری شده طبق بازخورد.

### تیم توسعه
- ارائه محدودیت های فنی (grid، responsive، performance، i18n، RTL/LTR).
- بازخورد دقیق، مرحله ای و اولویت بندی شده.
- پیاده سازی faithful به طراحی نهایی تایید شده.
- اعلام مواردی که نیاز به تعدیل بصری دارند (به دلیل UX یا performance).

### تصمیم گیر نهایی
- تایید نهایی بر پایه تطابق با نیاز کارفرما + امکان پیاده سازی.

---

## خروجی های اجباری طراح (Design Handoff Package)

طراح در هر مرحله باید این موارد را تحویل بدهد:

1. فایل منبع PSD
- لایه بندی کامل و نام گذاری شده
- ساختار گروه بندی روشن (Header, Hero, Sections, Footer, Components)

2. پیش نمایش تصویری
- PNG/JPG کامل صفحه Desktop
- PNG/JPG کامل صفحه Mobile
- در صورت نیاز Tablet

3. خروجی Assetها
- لوگوها، آیکون ها، تزئینات، پس زمینه ها
- ترجیحا WebP/SVG/PNG با نام گذاری استاندارد

4. Design Spec مختصر
- سایز فونت ها، وزن فونت، line-height
- رنگ ها (HEX)
- فاصله گذاری ها (padding/margin)
- حالت های دکمه و لینک (normal/hover/active)

5. لیست کامپوننت ها
- Button
- Card
- Section Title
- Form Field
- Navbar
- Footer
- Slider/Carousel block

6. نسخه بندی تحویل
- v1, v2, v3 ...
- changelog کوتاه برای هر نسخه (چه چیزی تغییر کرده)

---

## استاندارد طراحی برای اینکه پیاده سازی راحت شود

1. Grid و Layout
- Desktop بر اساس grid 12 ستونه
- Mobile تک ستونه یا دو ستونه کنترل شده
- حداکثر عرض محتوا برای desktop مشخص باشد (مثلا 1200 تا 1280)

2. Breakpointها
- Mobile: 360+
- Tablet: 768+
- Desktop: 1024+
- Large: 1280+

3. Typography
- سلسله مراتب واضح: H1, H2, H3, Body, Caption
- هر استایل فقط یک تعریف یکتا داشته باشد
- از تنوع غیرضروری فونت جلوگیری شود

4. Color System
- Primary / Secondary / Accent / Neutral / Success / Error
- کد HEX دقیق برای همه رنگ ها
- کنتراست قابل قبول برای متن ها

5. RTL/LTR
- طراحی فارسی در RTL
- برای انگلیسی/عربی تطبیق ساختار از ابتدا دیده شود
- فاصله ها و alignment وابسته به جهت متن باشند

6. تصاویر و Performance
- استفاده از تصاویر بهینه و قابل فشرده سازی
- اجتناب از فایل های بسیار حجیم در hero
- عناصر تزئینی تکرارشونده ترجیحا SVG یا asset سبک

## فرآیند همکاری (Workflow)

### مرحله 1: Brief مشترک
- نیاز کارفرما + رفرنس بصری + هدف صفحه
- تعریف KPI صفحه (مثلا کلیک روی رزرو نوبت)

### مرحله 2: Wireframe سریع
- اسکلت محتوا بدون جزئیات زیاد
- تایید ساختار سکشن ها و اولویت پیام ها

### مرحله 3: Design v1
- طراحی کامل Desktop + Mobile
- تحویل بسته کامل handoff

### مرحله 4: بازخورد کنترل شده
- بازخورد فقط در سه دسته:
  1) ضروری (Must)
  2) مهم (Should)
  3) سلیقه ای (Could)

### مرحله 5: Design v2 نهایی
- اعمال اصلاحات ضروری و مهم
- تایید نهایی برای شروع پیاده سازی

### مرحله 6: QA بعد از پیاده سازی
- مقایسه خروجی واقعی با طرح
- ثبت اختلاف ها و اصلاح نهایی

---

## قواعد بازخورد برای جلوگیری از اتلاف زمان

1. بازخورد پراکنده ممنوع
- بازخوردها تجمیع شده و یکجا ارسال شود.

2. بازخورد مبهم ممنوع
- به جای «قشنگ نیست»، نوشته شود:
  - سکشن: Hero
  - آیتم: CTA اصلی
  - مشکل: کنتراست کم
  - اصلاح پیشنهادی: رنگ پس زمینه دکمه تیره تر شود

3. هر بازخورد باید اولویت داشته باشد
- Must / Should / Could

4. تعداد چرخه اصلاح محدود
- ترجیحا حداکثر 2 چرخه اصلاح اصلی

## تحویل نهایی مورد انتظار از طراح

در هر مایلستون، یک پوشه تحویل شامل:
1. /source (PSD)
2. /preview (PNG/JPG)
3. /assets (SVG/PNG/WebP)
4. /spec (فایل متنی کوتاه مشخصات)
5. /changelog (تغییرات نسخه)

## نکات اجرایی ویژه این پروژه

1. تمرکز اصلی صفحه: تبدیل کاربر به رزرو نوبت.
2. CTA اصلی باید همیشه دیده شود (به خصوص در موبایل).
3. بخش های قبل/بعد، اعتمادسازی و معرفی کلینیک باید در روایت بصری مشخص باشند.
4. سبک بصری مرجع ارسالی رعایت شود، اما خروجی برای وب مدرن و سبک تر بهینه گردد.
5. طراحی باید کاملا قابل پیاده سازی در Next.js + Tailwind باشد.
