# Roadmap جامع و اجرایی برای تکمیل MVP کلینیک دکتر نیک

این نسخه، نسخه‌ی کاملاً اجرایی و قابل استفاده برای ادامه‌ی کار است.
هر گام به صورت اتمیک و قابل تست طراحی شده تا با کمترین خطا و بیشترین شفافیت پیش برویم.

---

## 1) قانون اجرایی
1. هر بار فقط یک گام اجرا شود.
2. بعد از هر گام، فقط این خروجی‌ها گزارش شود:
   - فایل‌های تغییر یافته
   - خلاصه‌ی تغییر
   - نتیجه‌ی typecheck
3. اگر گام دیتابیس دارد، migration جدا و قابل بازگشت ایجاد شود.
4. اگر خطا داشت، ابتدا همان خطا رفع شود، سپس به گام بعدی برویم.
5. برای هر گام، حداقل یکی از معیارهای پذیرش باید قابل تست باشد.

---

## 2) وضعیت فعلی پروژه

### انجام شده
1. مسیر رزرو و فرم بیمار در حالت عملیاتی است.
2. لینک «مشاهده نمونه کار پزشک» کنار فیلد پزشک اضافه شده است.
3. مسیر اختصاصی portfolio در حالتی اولیه وجود دارد.
4. داده‌ی mock portfolio و fallback route فعال هستند.
5. فیلتر گالری فقط روی approved در حال اجرا است.
6. پنل staff/admin برای تایید/رد مدیا در حالتی اولیه وجود دارد.
7. typecheck فعلی پاس می‌شود.

### مواردی که هنوز نیاز به تکمیل دارند
1. تکمیل داده و متن واقعی portfolio.
2. اتصال بهتر و پایدارتر لینک رزرو به مسیر واقعی پزشک.
3. انتقال و استانداردسازی فایل‌های واقعی از درایو F به مسیرهای پروژه (public/home، public/gallery، public/services و ...).
4. ویدئوهای واقعی و placeholder قوی‌تر برای portfolio.
5. محتوا و asset های نهایی برای صفحه اصلی و فوتر.
6. QA بصری و smoke نهایی برای deploy.

---

## 3) فازهای باقی‌مانده

### پیش‌نیاز اجرایی 0 — انتقال و استانداردسازی asset های واقعی از درایو F
هدف: فایل‌های واقعی کارفرما به صورت قابل استفاده و نگهداری‌شده وارد پروژه شوند تا placeholderها و مسیرهای موقت حذف شوند.

کارها:
1. فولدرهای اصلی از درایو F (مثلاً پنل‌های صفحه اول، خدمات تخصصی، سرویس، اینستاگرام و نتایج واقعی) را بررسی و دسته‌بندی کنیم.
2. assetهای مرتبط را به مسیرهای مناسب در پروژه منتقل کنیم؛ مثلاً: `public/home/`, `public/gallery/`, `public/services/` و `public/doctor/`.
3. هر عکس واردشده از فلش کارفرما قبل از اضافه شدن به مسیر نهایی، به فرمت استاندارد و حجم استاندارد تبدیل شود (مثلاً JPG/WebP برای عکس‌های تصویری و PNG فقط در صورت نیاز شفافیت؛ ابعاد مناسب برای استفاده در سایت و حجم معقول برای لود سریع).
4. نام‌گذاری ثابت و قابل فهم برای فایل‌ها تعریف کنیم (مثلاً `hero-main.png`, `doctor-portrait.png`, `trust-banner.png`).
5. مسیرهای فعلی UI را به assetهای جدید وصل کنیم و در صورت لزوم fallback را نگه داریم.

معیار پذیرش:
1. همه‌ی assetهای اصلی از درایو F در پروژه قابل دسترسی باشند.
2. مسیرهای UI به فایل‌های واقعی اشاره کنند و هیچ placeholder مهمی باقی نماند.
3. typecheck و build پاس شوند.

---

### فاز A — تکمیل مسیر portfolio و لینک پزشک

#### گام 1: تثبیت مسیر واقعی portfolio
هدف: اطمینان از اینکه لینک نمونه کار به مسیر واقعی و قابل پیش‌بینی اشاره کند.

کارها:
1. مسیر `/[locale]/doctor-portfolio/[doctorSlug]` را از نظر داده و fallback کامل‌تر کنیم.
2. اگر اسلاگ نامعتبر بود، صفحه‌ی 404 مناسب بازگردانده شود.
3. از داده‌ی map استاندارد برای نمایش اطلاعات استفاده شود.

معیار پذیرش:
1. route باز شود و 200 بدهد.
2. slug نامعتبر 404 بدهد.
3. typecheck پاس شود.

---

#### گام 2: اتصال لینک رزرو به portfolio واقعی
هدف: لینک کنار فرم رزرو بر اساس پزشک انتخابی و نه یک مسیر عمومی، مسیر درست را باز کند.

کارها:
1. doctorName را به slug قابل استفاده تبدیل کنیم.
2. href لینک نمونه کار را بر اساس doctorName و locale داینامیک کنیم.
3. اگر doctorName خالی بود، لینک غیرفعال شود.

معیار پذیرش:
1. تغییر نام پزشک، مسیر لینک را تغییر دهد.
2. حالت خالی لینک غیرفعال باشد.
3. typecheck پاس شود.

---

#### گام 3: تکمیل داده‌ی portfolio map
هدف: داده‌ها در یک منبع واحد متمرکز و قابل نگهداری باشند.

کارها:
1. فایل `src/lib/doctor-portfolio-map.ts` را با داده‌های کامل‌تر و قابل توسعه گسترش دهیم.
2. فیلدهای اضافی مثل `highlights`, `videoHints`, `ctaLabel` اضافه شود.
3. fallback برای پزشک‌های ناشناخته بهتر شود.

معیار پذیرش:
1. map از یک منبع واحد استفاده کند.
2. هیچ hardcode پراکنده‌ای در صفحه باقی نماند.
3. typecheck پاس شود.

---

#### گام 4: رندر حرفه‌ای‌تر صفحه portfolio
هدف: صفحه نمونه کار از صرف یک اسکلت، به یک صفحه‌ی قابل نمایش و قابل استفاده تبدیل شود.

کارها:
1. عنوان، توضیح، تصویر و بخش ویدئو را از داده‌ی map بخوانیم.
2. بخش placeholder ویدئو را به ساختاری بهتر و قابل توسعه تبدیل کنیم.
3. متن‌ها را چندزبانه و قابل فهم کنیم.

معیار پذیرش:
1. صفحه portfolio از داده‌ی map استفاده کند.
2. بخش ویدئو قابل جایگزینی با asset واقعی باشد.
3. typecheck پاس شود.

---

### فاز B — ویدئو و تایید مدیا

#### گام 5: زیرساخت ویدئو برای portfolio
هدف: پایه‌ی ویدئو برای آینده فراهم شود.

کارها:
1. مدل `Video` را کامل‌تر کنیم (اگر لازم باشد).
2. migration جدا برای ویدئو بسازیم.
3. یک سکشن placeholder ویدئو با ساختار قابل ارتقا در صفحه portfolio قرار دهیم.

معیار پذیرش:
1. migration قابل اجرا باشد.
2. صفحه portfolio در صورت نبود ویدئو، placeholder مناسب نمایش دهد.
3. typecheck پاس شود.

---

#### گام 6: تکمیل pipeline تایید مدیا
هدف: امکان تایید/رد عکس‌ها به‌صورت قابل اعتماد فراهم شود.

کارها:
1. وضعیت media را از `PENDING` تا `APPROVED/REJECTED` به‌طور کامل پایش کنیم.
2. مسیر API و UI مربوط به approve/reject را از نظر خطا و fallback بررسی کنیم.
3. پیام‌های موفقیت و خطا را در هر locale کنترل کنیم.

معیار پذیرش:
1. approve/reject از UI بدون خطای جزئی انجام شود.
2. وضعیت media به‌درستی به دیتابیس ذخیره شود.
3. typecheck پاس شود.

---

#### گام 7: گالری عمومی و approved-only
هدف: از انتشار محتوای نامناسب جلوگیری شود.

کارها:
1. query گالری عمومی را به‌طور دقیق روی `APPROVED` محدود کنیم.
2. در صورت خالی بودن گالری، پیام مناسب و قابل فهم نمایش داده شود.
3. fallback دیتابیس را با پیام خطای قابل فهم ترکیب کنیم.

معیار پذیرش:
1. آیتم pending/rejected در گالری عمومی دیده نشود.
2. حالت خالی گالری UX مناسب داشته باشد.
3. typecheck پاس شود.

---

### فاز C — تکمیل محتوا و asset های بصری

#### گام 8: جایگزینی placeholder های اصلی صفحه
هدف: بخش‌های مهم لندینگ را از حالت نمونه‌ای به حالت قابل ارائه نزدیک کنیم.

کارها:
1. asset‌های واقعی برای hero، doctor، instagram و trust را به UI وصل کنیم.
2. بخش‌های موجود که از placeholder استفاده می‌کنند، به نسخه‌ی نهایی نزدیک کنیم.
3. اگر asset واقعی در دسترس نیست، نسخه‌ی fallback با کیفیت مناسب استفاده شود.

معیار پذیرش:
1. صفحه اصلی بدون placeholder‌های مهم باقی نماند.
2. کیفیت تصویر و لود مناسب باشد.
3. typecheck پاس شود.

---

#### گام 9: تکمیل متن‌های اصلی و i18n
هدف: متن‌های واقعی و نه تستی در کل سایت قرار بگیرند.

کارها:
1. متن testimonial، faq، intro، footer و trust را بررسی و به‌روزرسانی کنیم.
2. اگر متن واقعی از کارفرما موجود نیست، حداقل یک نسخه‌ی قابل قبول و ثابت قرار دهیم.
3. کلیدهای جدید i18n را در fa/en/ar تکمیل کنیم.

معیار پذیرش:
1. متن‌های مهم در همه‌ی زبان‌ها قابل مشاهده باشند.
2. هیچ متن placeholder مهم در UI نماند.
3. typecheck پاس شود.

---

#### گام 10: polish نهایی و هماهنگی بصری
هدف: تجربه‌ی نهایی سایت شبیه به نسخه‌ی طراحی و قابل ارائه باشد.

کارها:
1. فاصله، رنگ، فونت، CTA و hierarchy را در مقیاس‌های مختلف بررسی کنیم.
2. بخش‌های کلیدی مثل هدر، فوتر، FAQ، trust و Instagram را با جزئیات نهایی هماهنگ کنیم.
3. در صورت نیاز، بخش‌های کم‌اهمیت را به صفحات داخلی منتقل کنیم.

معیار پذیرش:
1. طراحی کلی برای desktop/mobile قابل قبول باشد.
2. بخش‌های مهم از نظر UX و سئو واضح باشند.
3. typecheck پاس شود.

---

### فاز D — QA و آماده‌سازی deploy

#### گام 11: تست و smoke نهایی
هدف: اطمینان از عدم regression قبل از deploy.

کارها:
1. `npm run typecheck`
2. `npm run build`
3. smoke دستی برای booking، portfolio، gallery، staff media approval

معیار پذیرش:
1. همه‌ی چک‌ها سبز باشند.
2. هیچ خطای runtime مهم در مسیرهای اصلی وجود نداشته باشد.

---

#### گام 12: تنظیم نهایی برای production
هدف: سایت برای deploy آماده و پایدار باشد.

کارها:
1. تنظیمات env و مسیر deploy بررسی شوند.
2. reverse proxy / TLS / static asset path بررسی شوند.
3. خروجی نهایی برای انتشار آماده شود.

معیار پذیرش:
1. deploy با حداقل تنظیمات اضافی انجام شود.
2. مسیرهای اصلی در محیط واقعی قابل دسترس باشند.

---

## 4) ترتیب پیشنهادی اجرا
1. پیش‌نیاز 0 (انتقال و استانداردسازی asset های واقعی از درایو F)
2. گام 1 تا 4 (portfolio و لینک پزشک)
3. گام 5 تا 7 (ویدئو و تایید مدیا)
4. گام 8 تا 10 (asset و polish)
5. گام 11 تا 12 (QA و deploy)
