## ۵. بخش «قبل و بعد» — ⚠️ بازطراحی کامل لازم است

این بخش بزرگ‌ترین تفاوت طراحی است.

### وضعیت فعلی
- دو ردیف افقی ۵تایی از کارت‌ها (دندانپزشکی + پوست/فیلر) با اسلایدر چپ-راست.
- هر کارت یک تصویر است با برچسب فارسی زیر تصویر (ایمپلنت، کامپوزیت، …).
- داخل خود تصاویر «BEFORE / AFTER» تایپ شده.

### نسخه‌ی PSD (ردیف اول — دندانپزشکی)
- پنج تصویر هم‌اندازه کنار هم؛ تصویر وسط یک badge مشکی «BEFORE» و یک فلش دکمه‌ای روی آن دارد (suggesting hover swap to AFTER).
- زیر هر تصویر یک pill teal با عنوان (ایمپلنت | کامپوزیت | لمینت | بلیچینگ).
- دو فلش navigation سمت چپ و راست بیرون از قاب.
- در پایین یک نوار بزرگ‌تر با عنوان «قبل و بعدی» و یک فلش روی آن.

### نسخه‌ی PSD (ردیف دوم — پوست/فیلر)
- مشابه ردیف اول، ولی تصاویر هر کدام **به دو نیمه‌ی AFTER/BEFORE تقسیم شده** (نیم بالا AFTER، نیم پایین BEFORE روی همان عکس).
- برچسب‌ها: ژل چونه | فیلر بینی | فیلر لب | نخ سیلوئت.

### درخواست کارفرما (الهام از alluremedicalaesthetic.com)
بخش «قبل و بعد» باید مانند نمونه‌ی Allure باشد:
- چهار تصویر بزرگ افقی با لیبل‌های `20s | 30s | 40s | 50s` در پایین (نوار teal با متن سفید).
- روی هاور (یا کلیک در موبایل) روی هر گروه سنی، آن تصویر به یک panel با لیست خدمات/درمان‌های مناسب آن سن **منبسط/جایگزین** شود:
  - 20s: مثال — Preventive Skincare / Acne / …
  - 30s: مثال — Botox / Lip Filler / Hydrafacial
  - 40s: **Fine Tuning and Maintenance** — Radio Frequency Microneedling, ThermiVa (طبق اسکرین‌شات کارفرما)
  - 50s: مثال — Deep Resurfacing / Thread Lift / Volume Restoration
- پس‌زمینه‌ی panel سن باز شده آبی روشن (`#cfe3ea` تقریبی)؛ متن خدمت‌ها قهوه‌ای-سیاه با وزن متوسط.

### اتصال به Proxmox و LXC (برای deploy)

تمام کار از روی Windows با PowerShell + `plink`/`pscp` (PuTTY) انجام می‌شود.

| پارامتر | مقدار |
|--------|-------|
| Proxmox host | `192.168.85.7` |
| Proxmox user | `root` |
| Password | `Hs-co@12321#` (در env var `PLINK_PASS` ست می‌شود) |
| LXC container ID | `106` (نام: `clinic-lxc`) |
| LXC IP داخلی | `192.168.85.53` |
| پورت سایت | `3001` (port mapping LXC) |
| مسیر compose روی LXC | `/opt/dr_nik_clinic/` |
| مسیر اپ داخل کانتینر docker | `/workspace/clinic/` |

**الگوی فرمان از PowerShell:**

```powershell
$env:PLINK_PASS = 'Hs-co@12321#'
$cmd = 'pct exec 106 -- bash -lc "docker ps --format \"{{.Names}} | {{.Status}}\""'
plink -ssh -batch -pw $env:PLINK_PASS root@192.168.85.7 $cmd
```

**آپلود فایل (pscp → pct push → docker cp):**

```powershell
# 1) از ویندوز به Proxmox host
pscp -batch -pw $env:PLINK_PASS .\my-bundle.tar.gz root@192.168.85.7:/tmp/

# 2) از Proxmox host به داخل LXC
plink -ssh -batch -pw $env:PLINK_PASS root@192.168.85.7 'pct push 106 /tmp/my-bundle.tar.gz /tmp/my-bundle.tar.gz'

# 3) داخل LXC، استخراج و کپی به کانتینر docker
plink -ssh -batch -pw $env:PLINK_PASS root@192.168.85.7 'pct exec 106 -- bash -lc "cd /tmp && tar xzf my-bundle.tar.gz && docker cp ./bundle/. dr_nik_clinic_app:/workspace/clinic/"'
```

**نکته‌های مهم PowerShell + plink:**

- **همیشه از single-quote برای `$cmd`** استفاده کنید؛ double-quote باعث می‌شود PowerShell متغیرها/`$(...)` را قبل از ارسال جایگزین کند و فرمان شل لینوکس را خراب کند.
- اگر در فرمان Linux نیاز به `$()` یا `$variable` بود، باید `\$` بنویسید یا کلاً interpolation را به Linux بسپارید.
- `Invoke-WebRequest` در PowerShell 7 در حالت خطا `HttpResponseMessage` می‌دهد (نه `GetResponseStream`)؛ در `catch` از API هم‌خوان با PS7 استفاده کنید.
- `scp -r dir host:/remote/dir/` آن را به `/remote/dir/dir/` تو در تو می‌سازد؛ از `dir/.` به‌جای `dir` استفاده کنید.

### رجیستری ایران برای `npm install`

دلیل: لینک مستقیم به `registry.npmjs.org` از داخل سرور (LXC) ممکن است کند/غیرپایدار باشد. ولی mirror ایران **همیشه کامل نیست**.

**ترجیح اول (سریع‌تر):** `npm.iranserver.com`

```bash
# داخل LXC یا Dockerfile.patch:
npm config set registry https://npm.iranserver.com/
npm ci --include=dev
```

**Fallback ضروری** اگر بسته‌ها پیدا نشد:

- بسته‌های شناخته‌شده‌ی ناقص روی `npm.iranserver.com`: `@trpc/server`, `@prisma/client`, `next`, `react`, `zod` (طبق memory).
- در صورت خطای `404` یا `tarball not found`، رجیستری را موقتاً به npm رسمی بازگردانید:
  ```bash
  npm config set registry https://registry.npmjs.org/
  npm ci --include=dev
  ```
- یا از `--registry` فقط برای آن دستور:
  ```bash
  npm ci --include=dev --registry=https://registry.npmjs.org/
  ```

**در `Dockerfile.patch` چه می‌کنیم؟**

اگر patch build لازم نشد `npm install` بزند (چون از image قبلی node_modules را می‌برد) نیازی به این تنظیم نیست. ولی اگر بسته‌ی جدیدی در `package.json` اضافه شد، **حتماً `npm ci --include=dev` در Dockerfile.patch بزنید** (نه COPY از stale base) و رجیستری را با fallback ست کنید:

```dockerfile
RUN npm config set registry https://npm.iranserver.com/ && \
    (npm ci --include=dev || \
     (npm config set registry https://registry.npmjs.org/ && npm ci --include=dev))
```

### Node.js version روی سرور

- `next build` با Tailwind v4 روی **Node 18** با خطای `Cannot find native binding` از `@tailwindcss/oxide` fail می‌شود.
- داخل image، Node باید **20 یا بالاتر** باشد (base image فعلی این را تأمین می‌کند).
- اگر Dockerfile.patch مستقیماً روی LXC build می‌شود نه از داخل image، مطمئن شوید Node 20+ نصب است: `node -v`.
